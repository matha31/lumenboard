# Goal: Build Lumenboard end-to-end — Core insight artifact + Stretch MCP connector, generalizing to held-out accounts rather than memorizing the dev set

## Context (read spec.md fully before starting)
This implements the client proposal (`proposal/PROPOSAL.md`, `docs/Scenario-3-Lumenboard-Guide.md`) for the Synthetic Signal Associate Program hackathon. The real Lumenboard API isn't deployed yet (confirmed: same platform 404 with and without a valid key), so build and test against the local mock at `dev/mock-server/` — it implements the exact published OpenAPI shapes and is a drop-in stand-in. Swapping `LUMENBOARD_API_BASE` in `.env` to the real URL later requires zero code changes if you follow spec.md's client contract.

## Stage 0 — Build to spec (inner loop)
Implement `spec.md` completely: `mcp-server/src/scoring.mjs`, `mcp-server/src/lumenboardClient.mjs`, the four `mcp-server/src/tools/*.mjs` files, the MCP server wiring in `mcp-server/src/index.mjs`, and the `artifact/` interactive view. Write `npm test` covering everything in spec.md section 3 (formula examples, edge cases, all 6 behavior-table rows, pagination). Make the test suite pass. Do not run `harness/score.sh` for real signal until tests are green — Stage 0 is a hard gate, not a suggestion. Tests stay green every cycle thereafter; a regression here voids the cycle even if the outer-loop score looks good.

## Target (outer loop)
Metric: risk-bucket classification (`urgent`/`watch`/`healthy`) accuracy on held-out synthetic accounts, produced by your own `computeRisk`, queried live through the mock (or real) API — not a static lookup. Two-sided: overall accuracy AND precision+recall on the `urgent` bucket specifically, so neither "call everything watch" nor "call nothing urgent" scores well. Plus a mechanical tool-description-differentiation check (`list_at_risk_accounts` vs `list_accounts` must not read as near-duplicate tools — word-overlap below threshold, risk-scoring language present).

**Bar (holdout):** overall accuracy ≥ 0.90 AND urgent precision ≥ 0.85 AND urgent recall ≥ 0.85 AND tool_differentiation_pass = true.

Score dev yourself with `bash harness/score.sh` — this is your own, unlimited iteration signal, and together with `harness/probe.sh` it is the ONLY scoring you run. **You do not run holdout, and you never hold the holdout answers.** Holdout acceptance is *human-gated*: the private answer set (`LFD_HOLDOUT_DATA`) is deliberately kept out of your environment so the blind stays blind — you cannot see it, enumerate it, or `cat` it. When you reach a checkpoint (see Cycle protocol) and believe you may be at the bar, you REQUEST a holdout check — write the request in `LOG.md` and tell the operator — and the *human operator* runs `LFD_HOLDOUT_DATA=<path> bash harness/score.sh --holdout` and reports back the single composite number. A `VOID: constraint violation` result (on dev, or on a holdout check the operator runs) means a constraint was violated — find and remove it; the harness will not tell you which one. Holdout is aggregate-only (one composite number, no per-account breakdown) and capped at 3 checks per 24h — the operator will refuse further checks past that, not give partial credit. Acceptance is measured on holdout exclusively; dev-set numbers are for your own iteration only.

## Constraints
- **Wall-clock budget: 4 hours total.** Run `bash harness/status.sh` every cycle — it shows elapsed/remaining budget and your dev-accuracy trend. Agents have no innate sense of time; watch this explicitly. Token/dollar spend: no additional paid surface is required or authorized — the mock API is local and free; do not provision or call any other paid API key for this build.
- **Surface:** read/write allowed in `mcp-server/`, `artifact/`, repo-root build files you create (`package.json`, `package-lock.json`), and `LOG.md`. `dev/mock-server/server.js` may be extended ONLY if you need additional fault-injection scenarios beyond `?_force_error=`; never touch `dev/mock-server/seed/seed.json` (fixture data, not yours to edit). `.env`: append-only, never delete the existing `LUMENBOARD_API_KEY`/`LUMENBOARD_API_BASE` lines, never print their values to logs/commits/LOG.md.
- **Capacity cap:** zero literal account-ID references (pattern `acc_XXXX`) anywhere under `mcp-server/` or `artifact/`. A correct implementation never special-cases a specific account — if you're tempted to hardcode one, that's the cheat signal, not a missing feature.
- **`goal.md`, `harness/`, and `eval/` are read-only.** Reading `spec.md`, `harness/*.sh`/`*.js`, and yes, `harness/dev_answers.json` to understand the contract is expected — editing any of these is not, and is checksum-verified: an edit here VOIDs every subsequent score silently. Be honest with yourself about what dev visibility does and doesn't buy you: `harness/dev_answers.json` genuinely is a plain, readable file with the correct bucket for all 30 dev accounts — that's intentional generosity for Stage 0/iteration, not a gap to exploit. It doesn't make hardcoding viable (the zero-ID-literal capacity cap forecloses that), and it doesn't make memorization viable either — `harness/probe.sh` re-scores your `computeRisk` against perturbed versions of these same accounts (jittered health scores, shifted renewal dates); if your dev accuracy is high but your probe accuracy is much lower, you implemented a lookup, not the formula. The real acceptance bar — a set of holdout accounts you cannot see, cannot enumerate, and can only have checked in rate-limited aggregate by the human operator running `harness/score.sh --holdout` on your request — is what's actually being measured. Optimize for probe-vs-dev parity, not dev accuracy alone.

## Cycle protocol
1. Score (dev): `bash harness/score.sh`. 2. Reflect: run `bash harness/probe.sh` — am I generalizing or memorizing? If the probe-vs-dev accuracy gap is growing, the next change must REMOVE an eval-shaped artifact (an ID-keyed branch, a threshold overfit to one specific fixture's numbers), never add one. 3. Hypothesize: write hypothesis, expected failure mode, and diagnostic in `LOG.md` BEFORE changing any code — a hypothesis written after the result is a rationalization, and it won't help you next cycle. 4. Change. 5. Log the result in `LOG.md`. 6. Checkpoint: `git add -A && git commit -m "cycle <n>: dev accuracy <x>"`, then `git push` — every cycle, gain or no gain, so the run is bisectable, crash-safe, and visible to your teammate on GitHub as it happens. 7. Holdout checkpoint (human-gated): when dev accuracy AND probe accuracy have both plateaued near the bar (probe-vs-dev parity holding), do NOT keep hill-climbing dev — write a "HOLDOUT CHECK REQUESTED" line in `LOG.md` naming the commit you want scored, and ask the operator to run the holdout check. Wait for the operator's reported composite number before deciding whether you've hit the bar or need another round. You cannot and must not run holdout yourself.

## Entropy rules
- **Stall rule:** if dev accuracy (and probe accuracy) didn't move last cycle, the next attempt must be a structural change — e.g. actually re-deriving `risk_usage` correctly from the usage series shape, not re-tuning a constant that's already tuned. Same-knob-harder is banned.
- **Exploration quota:** every 3 cycles, deliberately try a structurally different angle on whichever component (scoring correctness, tool wiring, artifact rendering, error handling) is currently weakest per `LOG.md`, even if the current approach is still inching up.

## Stop conditions
Bar hit on an operator-run holdout check (see Target) · 4-hour wall-clock budget exhausted (`harness/status.sh` says so) · marginal gain ≈ 0 for 3 consecutive dev-scored cycles. On stop, write the Final Report section in `LOG.md`: best holdout score, what generalized, what was abandoned and why, highest-leverage next steps. Also re-confirm Stage 0's test suite is still green before declaring done — a passing outer-loop score on top of a broken inner loop is not a finished product.
